- Investigated unstable OpenClaw app-style launcher on Windows (`C:\Users\30698\OneDrive\Desktop\random files\OpenClaw.vbs`) that runs `wsl -e bash /home/user/start-openclaw.sh`.
- Found original launcher issue: `start-openclaw.sh` manually spawned gateway + Mission Control in background and used `wait`, creating fragile process ownership and duplicate runtime conflicts.
- Reworked startup model toward systemd user services:
  - `~/.config/systemd/user/openclaw-mission-control.service` created for Mission Control using direct `next start -H 127.0.0.1 -p 3000`.
  - `start-openclaw.sh` changed multiple times to use systemd-managed services, logging, and URL readiness checks.
- Important root-cause findings:
  - Gateway itself often stayed healthy and returned `HTTP 200` on `http://127.0.0.1:18789` even when user perceived it as crashed.
  - Many failures were caused by duplicate/manual processes on ports 18789/3000 and by the launcher using `restart`, causing service churn and web UI disconnects.
  - Mission Control repeatedly hit `EADDRINUSE` when stale/manual `next-server` processes were still holding port 3000.
  - Service unit bug fixed by replacing `npm run start -- -H 127.0.0.1 -p 3000` with direct `.../node_modules/.bin/next start -H 127.0.0.1 -p 3000`, so systemd tracks the real `next-server` process.
- Config work:
  - Restored `~/.openclaw/openclaw.json` from `openclaw.json.bak` after suspected config/schema drift.
  - Removed disabled `plugins.entries.acpx` config block to stop warning spam about disabled-but-configured plugin.
- Logging added:
  - `/home/user/.openclaw/logs/start-openclaw.log` records launcher actions, readiness, URL opening decisions.
- Current diagnosis before compaction:
  - The double-click app experience is still flawed because launcher behavior is too aggressive; it should `start if inactive` rather than `restart` services on every launch.
  - Evidence also showed earlier WSL/user-session churn (different systemd user instances / boot boundaries), but latest state had both services active and responsive.
  - Next recommended step: change launcher from forced `restart` to `start if inactive`, then re-test browser/webchat transport behavior.
